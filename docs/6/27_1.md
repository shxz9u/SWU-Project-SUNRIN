## 1. 아키텍처 및 평가 설계

### 1.1 목적

본 단계의 목적은 실물 Raspberry Pi가 없는 상황에서도 IoT 펌웨어/C 코드 취약점 분석 파이프라인을 검증할 수 있도록 전체 아키텍처와 평가 기준을 설계하는 것이다.

기존 계획에서는 Raspberry Pi를 분석 게이트웨이로 사용하려 했지만, 현재는 실물 장비가 없으므로 Linux 호스트 PC와 Docker 컨테이너를 이용해 Pi의 역할을 대체한다. 호스트 PC에서는 Ollama 기반 Gemma4 모델을 실행하고, Docker 컨테이너 내부에서는 C 코드 또는 바이너리를 대상으로 LLM 분석, fuzzing, 실행 검증을 수행한다.

본 프로젝트에서 LLM은 취약점을 최종 판정하는 도구가 아니라, 취약점 후보와 fuzzing seed를 생성하는 보조 분석기로 사용된다. 최종 취약점 여부는 fuzzing과 재현성 검증 결과를 기준으로 판단한다.

---

### 1.2 전체 아키텍처

```text
[Linux Host PC]
├── Ollama
│   └── Gemma4 모델 실행
│
└── Docker Container
    ├── 분석 대상 C 코드 / 바이너리 입력
    ├── LLM 기반 취약점 후보 분석
    ├── fuzz seed 생성
    ├── mutation fuzzing 실행
    ├── crash / timeout / sanitizer error 수집
    ├── 동일 입력 재실행을 통한 재현성 검증
    └── 최종 결과 저장
```

호스트 PC는 LLM 추론을 담당하고, Docker 컨테이너는 분석 파이프라인 실행 환경으로 동작한다. 컨테이너는 호스트의 Ollama API에 요청을 보내 C 코드 분석 결과와 fuzzing seed를 생성하고, 이후 해당 seed를 기반으로 대상 바이너리에 입력을 전달하여 실제 이상 동작 여부를 확인한다.

---

### 1.3 데이터 흐름

```text
1. 분석 대상 입력
   - C 소스코드
   - 실행 바이너리
   - 설정 파일
   - 샘플 입력

        ↓

2. LLM 기반 취약점 후보 분석
   - 위험 함수 탐지
   - 입력 검증 부족 확인
   - 메모리 안전성 문제 추론
   - 취약점 후보 생성
   - fuzzing seed 추천

        ↓

3. Fuzzing 입력 생성
   - LLM이 제안한 seed 사용
   - 길이 변형
   - 특수문자 삽입
   - 경계값 입력 생성
   - 랜덤 변이 입력 생성

        ↓

4. 실행 기반 Fuzzing
   - 대상 바이너리 반복 실행
   - 변이 입력 전달
   - crash / timeout / error 수집

        ↓

5. 검증 단계
   - 문제가 발생한 입력 재실행
   - 동일 증상 반복 여부 확인
   - sanitizer 로그 확인
   - 재현성 판단

        ↓

6. 최종 결과 분류
   - CONFIRMED
   - FUZZ_TRIGGERED
   - NOT_REPRODUCED
   - FALSE_POSITIVE
```

---

### 1.4 주요 구성 요소

| 구성 요소            | 역할                             |
| ---------------- | ------------------------------ |
| Linux Host PC    | Ollama와 Gemma4 모델을 실행하는 환경     |
| Docker Container | 분석 파이프라인이 실행되는 격리 환경           |
| Ollama API       | 컨테이너에서 LLM 분석 요청을 보내는 API      |
| LLM Analyzer     | C 코드에서 취약점 후보와 fuzz seed를 생성   |
| Seed Generator   | LLM 결과를 기반으로 fuzzing 입력 생성     |
| Simple Fuzzer    | 대상 바이너리에 변이 입력을 반복 전달          |
| Verifier         | crash 또는 error 발생 입력을 재실행하여 검증 |
| Result Writer    | JSON 및 Markdown 형식으로 결과 저장     |

---

### 1.5 LLM 분석 단계

LLM 분석 단계에서는 C 코드 또는 펌웨어 분석 결과를 입력으로 받아 취약점 후보를 생성한다. 이 단계의 결과는 최종 취약점이 아니라 검증이 필요한 후보로만 취급한다.

LLM이 중점적으로 탐지할 취약점 유형은 다음과 같다.

```text
- Buffer Overflow
- Unsafe Function Usage
- Format String Bug
- Command Injection Risk
- Hardcoded Credential
- Missing Input Validation
- Memory Safety Issue
```

LLM 출력 예시는 다음과 같은 구조를 가진다.

```json
{
  "candidate_id": "LLM-001",
  "vulnerability_type": "buffer_overflow",
  "file": "input.c",
  "function": "parse_config",
  "reason": "strcpy is used without length validation",
  "suggested_seeds": [
    "normal",
    "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA",
    "%x%x%x%x"
  ],
  "confidence": "medium",
  "status": "LLM_CANDIDATE"
}
```

---

### 1.6 Fuzzing 및 검증 단계

Fuzzing 단계에서는 LLM이 생성한 취약점 후보와 seed를 기반으로 실제 대상 바이너리에 다양한 입력을 전달한다. 이때 crash, timeout, sanitizer error와 같은 실행 기반 증거를 수집한다.

검증 단계에서는 fuzzing 중 문제가 발생한 입력을 동일 조건에서 여러 번 재실행한다. 동일 입력으로 문제가 반복적으로 발생하는 경우에만 최종 취약점으로 분류한다.

```text
LLM_CANDIDATE
→ LLM이 취약점 후보를 생성한 상태

FUZZ_TRIGGERED
→ fuzzing 중 crash, timeout, sanitizer error 등 이상 동작이 발생한 상태

CONFIRMED
→ 동일 입력 재실행에서 문제가 재현된 상태

NOT_REPRODUCED
→ fuzzing 중 이상 동작이 있었지만 재실행에서 재현되지 않은 상태

FALSE_POSITIVE
→ LLM은 의심했지만 실행 검증에서 문제가 확인되지 않은 상태
```

---

### 1.7 최종 판정 기준

본 프로젝트에서는 LLM 분석 결과만으로 취약점을 확정하지 않는다. 최종 취약점 판정은 실행 기반 검증 결과를 기준으로 수행한다.

판정 로직은 다음과 같다.

```text
1. LLM이 취약점 후보를 생성한다.
   → 상태: LLM_CANDIDATE

2. Fuzzing 중 crash, timeout, sanitizer error가 발생한다.
   → 상태: FUZZ_TRIGGERED

3. 동일 입력으로 3회 재실행했을 때 2회 이상 동일 문제가 발생한다.
   → 상태: CONFIRMED

4. 재실행에서 문제가 반복되지 않는다.
   → 상태: NOT_REPRODUCED

5. 실행 검증에서 문제가 전혀 확인되지 않는다.
   → 상태: FALSE_POSITIVE
```

### 1.8 평가 지표

본 실험에서는 LLM 기반 분석과 fuzzing 기반 검증의 효과를 비교하기 위해 다음 평가 지표를 사용한다.

| 평가 지표     | 설명                        | 측정 방법                         |
| --------- | ------------------------- | ----------------------------- |
| 탐지율       | 실제 취약점 중 LLM이 후보로 탐지한 비율  | `LLM 후보 수 / 정답셋 취약점 수`        |
| 검증 성공률    | LLM 후보 중 실제 실행 검증을 통과한 비율 | `CONFIRMED 수 / LLM 후보 수`      |
| 오탐률       | LLM 후보 중 실제 검증되지 않은 비율    | `FALSE_POSITIVE 수 / LLM 후보 수` |
| 소요 시간     | 전체 분석에 걸린 시간              | LLM 분석 + fuzzing + 검증 시간      |
| 재현 성공 수   | 동일 입력 재실행으로 확인된 문제 수      | CONFIRMED 상태 개수               |
| crash 수   | fuzzing 중 발생한 비정상 종료 수    | crash 로그 개수                   |
| timeout 수 | 특정 입력에서 제한 시간을 초과한 횟수     | timeout 로그 개수                 |

---

### 1.9 실험 결과 저장 구조

분석 결과는 추후 비교와 보고서 작성을 위해 JSON과 Markdown 형식으로 저장한다.

```text
project/
├── Dockerfile
├── docker-compose.yml
├── analyzer/
│   ├── main.py
│   ├── ollama_client.py
│   ├── prompt_builder.py
│   ├── seed_generator.py
│   ├── simple_fuzzer.py
│   ├── verifier.py
│   └── result_writer.py
│
└── work/
    ├── target/
    │   ├── input.c
    │   └── target_binary
    │
    ├── seeds/
    │   └── seed_001.txt
    │
    ├── crashes/
    │   └── crash_001.input
    │
    └── logs/
        ├── llm_candidates.json
        ├── fuzz_result.json
        ├── verification_result.json
        └── final_report.md
```

각 로그 파일의 역할은 다음과 같다.

| 파일명                        | 내용                 |
| -------------------------- | ------------------ |
| `llm_candidates.json`      | LLM이 생성한 취약점 후보 목록 |
| `fuzz_result.json`         | fuzzing 실행 결과      |
| `verification_result.json` | 재현성 검증 결과          |
| `final_report.md`          | 최종 분석 요약 보고서       |

### 1.10 현재 단계의 범위

현재 단계에서는 실제 펌웨어 대상을 확정하지 않고, 분석 파이프라인의 구조와 평가 기준을 먼저 설계한다. 초기 구현은 샘플 C 코드와 로컬 바이너리를 대상으로 진행하며, 이후 DVRF, IoTGoat, 실제 오픈소스 펌웨어 등으로 분석 대상을 확장할 수 있도록 설계한다.

현재 구현 범위는 다음과 같다.

```text
[포함]
- Linux Host 기반 Ollama Gemma4 실행
- Docker 기반 분석 컨테이너 구성
- LLM 기반 취약점 후보 생성
- LLM 기반 fuzz seed 생성
- 간단한 mutation fuzzing
- crash / timeout / sanitizer error 수집
- 재현성 검증
- 최종 결과 보고서 생성

[추후 확장]
- 실제 IoT 펌웨어 분석
- QEMU 기반 펌웨어 에뮬레이션
- AFL++ 연동
- coverage 기반 fuzzing
- crash minimization
- 다양한 C 오픈소스 프로젝트 적용
```

### 1.11 1단계 산출물

본 단계의 산출물은 다음과 같다.

```text
[1] LLM → Fuzzing → 검증 기반 전체 아키텍처 설계
[2] Linux Host + Docker 기반 Pi 대체 구조 정의
[3] 데이터 흐름도 작성
[4] LLM 분석 결과를 최종 판정으로 사용하지 않는 검증 기준 수립
[5] 탐지율, 검증 성공률, 오탐률, 소요 시간 등 평가 지표 정의
[6] 결과 저장 파일 구조 정의
```

이 설계를 기반으로 다음 단계에서는 Docker 컨테이너와 Ollama API 연동을 구현하고, 샘플 C 코드를 대상으로 LLM 분석 및 fuzzing 검증 파이프라인을 구축한다.
